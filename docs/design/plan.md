## Problem Statement

Adir is A1 in Dutch, pushing toward A2 in the near term and B1 further out. He has a 750-day Duolingo streak, so reading is fine, but his listening and speaking are weak — the two hardest skills to practice alone. He tried ChatGPT/Claude voice mode and a custom GPT to close this gap, and both failed the same way: no memory across sessions, no curriculum structure, generic responses, no habit-forming behavior. The effort to make any session useful is lost the next time he opens the app.

The app he actually wants is a mobile web app "talking buddy" that:

- Remembers him session to session — what he struggled with, what he's mastered, what's overdue for review
- Covers an A1→A2→B1 curriculum explicitly, starting where he actually is (A1 solidification + A2 push), with the schema extensible to B2+ later
- Feels like a patient friend, not a quiz — real-feeling micro-conversations, gentle corrections, paced to his level
- Defaults to immersive Dutch, with an English escape hatch he controls
- Fits into a daily 5-10 minute slot (replaces the Duolingo habit slot, doesn't add a second one)

The starting codebase is a forked English↔Hebrew word-quiz app (Node/Express, single HTML page, Web Speech API, `.docx` uploaded word lists). Most of it will be deprecated. Only the deployment intuition (Node + nginx) carries forward.

## What Makes This Cool

The thing that makes this different from every existing AI language app (Talkpal, Loora, Speak, plain ChatGPT voice) is that **memory is the product**. Not the voice. Not the curriculum. Memory.

Every existing app is Groundhog Day — you open it, it doesn't know you, you have the same kind of conversation you had last week. This one compounds. Every session makes the next session sharper. The magic moment on day 2 is the buddy saying "last time you mixed up the modal verb word order twice — kunnen/willen/moeten send the second verb to the end of the sentence. Let's hit that pattern again in a new conversation." That sentence is the whole product. The voice is just the I/O layer that delivers it.

The architecture that makes this possible isn't novel — it's the classical Intelligent Tutoring Systems design from 1990s AI-in-education research, just wearing 2026 voice clothes: **persistent learner model + session planner + voice conversation driver + retrospective learner-model update**. Three coordinating systems, not one giant prompt. Consumer AI voice apps can't do this because an LLM context window is not durable structured memory. A real app can.

## Constraints

- **Personal use, but public URL.** Single user for the foreseeable future, but the app sits on a public Vercel subdomain with paid API calls. Auth is required as a cost gate, not for multi-tenancy. Passwordless magic link, hardcoded to Adir's email.
- **Mobile-first web app.** Target: iOS Safari and Android Chrome. PWA-installable eventually, but even a bookmarked web page is fine for V1.
- **Daily 5-10 minute sessions.** Not a marathon tool. Replaces a Duolingo slot.
- **Budget: ~$5/month ceiling for APIs.** Personal tool, not a business. Should be comfortably under.
- **Adir's time is limited.** He's a fractional CMO with client work. Build in short focused sessions, not long heroic sprints.
- **Start with the existing repo, but don't be precious about it.** Most of it gets deleted. The only thing that matters is that the new thing ships and works.

## Premises

1. **The real product is a persistent learner model + voice-conversation driver.** Voice is I/O. Memory lives outside the LLM. Nail memory + curriculum and a mediocre voice still delights; nail voice and ship with amnesia and it dies in a week.

2. **Curriculum is baked in for V1, but the schema is CEFR-agnostic and extensible.** V1 seeds A1→A2 content (NT2 topics + grammar concepts + core vocab at the user's actual level), with a handful of B1-adjacent items available once confidence on the A2 core is earned. Adding more B1 content, then B2, is a content-loading session later — not a rewrite. The old `.docx` upload mechanic dies.

3. **Paid voice from day one.** Deepgram Nova-2 Dutch for STT (excellent Dutch accuracy), OpenAI TTS `nova` for TTS (human-quality voice, streaming support). The cross-model second opinion flagged mobile Safari Dutch STT as mediocre, and for a tool used daily for years, free Web Speech is false economy. Cost breakdown below.

4. **The LLM is the conversation brain, but the app owns its memory.** Every turn, the app assembles a curated system prompt: learner profile + today's target + recent weak spots + last 6 turns + anti-repetition window. The LLM doesn't hold state; the app does. This is the architectural choice that makes "learn and adapt" real.

5. **Sessions are short and daily.** 5-10 minute default. The habit engine is already proven by his Duolingo streak. This app replaces that slot, doesn't add a second one.

6. **The highest-risk unknown is the learner-model loop, not the voice pipeline.** Voice is a solved commodity now. The real question is: does "the app remembered me" feel good after 7 real days of use? That's what V1 tests.

## Cross-Model Perspective

The Claude subagent, reviewing the problem cold, agreed with the architecture thesis and sharpened two things:

- **The verbatim signal:** the user said, "the effort to turn it into something useful is going to be lost the next time I open it." The subagent read this correctly as the real pain point — discontinuity, not voice quality. Design implication: memory must be **legible** in the UI. On open, show him what the buddy remembered about yesterday, in plain English. That's the dopamine hit.

- **The V2 flagship feature: Eavesdropping mode.** Buddy role-plays a 15-30 second scene between two Dutch speakers (two neighbors gossiping, a shopkeeper and customer haggling, a doctor and patient). User listens silently, then Buddy asks in Dutch: "what did they argue about? how did the shopkeeper feel?" This directly attacks the listening gap better than conversation does, because real Dutch-in-the-wild is overhearing fast native speech between locals at Albert Heijn, not people talking slowly to you. Premature for V1, locked in as V2.

- **Challenged premise #3** (Web Speech API is enough). The challenge stuck — premise revised above.

The subagent's prescription for the weekend build matched Approach A almost exactly, with one different call (Pipecat for voice pipeline vs. direct API calls from Next.js). For a single-user personal tool without barge-in requirements in V1, direct API calls are simpler. Pipecat moves in later if/when barge-in becomes the limiting factor.

## Approaches Considered

### Approach A: The Remembering Loop (minimal viable)
Ship the smallest thing that proves the memory thesis. ~16-24 hours of focused build time, spread across 4-6 short evening/weekend sessions (not one heroic sprint). Deprecate the old Node/Express app. Next.js on Vercel, SQLite via Turso or file, Deepgram + OpenAI/ElevenLabs voice, Claude Sonnet 4.6 driver. One screen: tap-to-talk, transcript, "what Buddy noticed" panel. 20 seeded A1/A2 items (weighted toward A1 consolidation with A2 stretch). Single learner-model table. Post-turn analyzer updates memory. Magic moment on next open: "last time you stumbled on X. Today: X again, new context." Explicitly NOT in V1: topic switching, English escape hatch, SRS math (stub), eavesdropping, accounts, progress heatmaps, session-length options.

### Approach B: The Adaptive Tutor (ideal architecture)
All of A plus: curriculum graph with prerequisites, real FSRS spaced repetition, multi-dimension weak-spot detection, "buddy proposes, you confirm" home flow, session-length options, English escape hatch, legible progress (heatmap by topic, concept-mastery streak). 3-4× the effort. Risk of over-engineering before validating the core loop.

### Approach C: Listening-First with Eavesdropping mode (creative / lateral)
Flip the script. Start with comprehension drills via generated two-speaker dialogues. Speaking buddy added later. Attacks listening gap directly. Contradicts the user's stated "talking buddy first" vision.

## Recommended Approach

**Approach A: The Remembering Loop.**

The single feature that has to work is: "it remembered me." Until that works, no amount of curriculum elegance or voice polish matters. A ships in 16-24 focused hours, gets used for a real week, generates the signal we actually need: does the Remembering Loop feel good? If yes, B is the roadmap. If no, we learn faster with less sunk cost. Approach C is too good to skip entirely — Eavesdropping mode becomes the V2 flagship feature.

### Tech stack (committed)

- **Framework:** Next.js 15 on Vercel, App Router. Route Handlers (NOT Server Actions) for streaming. Vercel Fluid Compute enabled (default on new projects) + 7am cron hitting `/api/health` to mitigate cold-start latency on the morning session.
- **Auth:** passwordless magic link, hardcoded allowlist of Adir's email. Auth.js (NextAuth v5) email provider + Resend for delivery. Session cookie, `middleware.ts` protects all `/api/*` except magic-link callbacks. This is the API-cost gate, not a multi-user feature.
- **DB:** SQLite via [Turso](https://turso.tech/) and `@libsql/client`. Required (not optional) — Vercel is stateless so local SQLite files don't persist. Turso's free tier covers this use case comfortably.
- **STT:** [Deepgram Nova-2 nl](https://deepgram.com/). **Browser → Deepgram direct** over WSS, using a 30-second JWT minted by `/api/deepgram/token` (Vercel functions can't proxy long-lived sockets on hobby tier). Opus-in-WebM from MediaRecorder (supported on mobile Safari 15+).
- **TTS:** OpenAI TTS `nova` voice with streaming enabled, proxied via a `ReadableStream` Route Handler. ~16KB audio per reply, well under bandwidth budget.
- **LLM:** Claude Sonnet 4.6 as both conversation brain and post-session analyzer. **Pre-build spikes required (before any code):**
  1. Dutch A1/A2 register quality: run 10 Dutch A1/A2 prompts through Sonnet 4.6. Verify output uses simple sentences, high-frequency vocab, present + perfectum mostly, minimal bijzin. If not acceptable, fall back to GPT-4o or evaluate Gemini.
  2. **STT accuracy on actual accented A1 voice** (added after eng review): 30 minutes in Deepgram's playground with 10 hand-picked A1/A2 Dutch sentences in your own voice. Compute WER vs ground truth. If WER > 25%, voice-stack revision is needed BEFORE building (try Whisper-3-large, try better speaker adaptation, or pivot toward shadowing-first design).
- **Deploy:** Vercel. Auto-deploy on push to main. Existing Node/Express + nginx setup retired.

### Latency budget

Target: <3 seconds from user-finishes-speaking to buddy-starts-speaking. Break down:
- Deepgram STT: ~300ms (streaming, final result shortly after silence)
- Claude Sonnet 4.6 driver call: ~800-1500ms (first token latency)
- OpenAI TTS streaming: <500ms to first audio chunk
- **Total first-audio latency: ~1.6-2.3s** — acceptable for a conversation feel.

If this exceeds 3s in practice, investigate: (a) is OpenAI TTS streaming actually streaming, (b) reduce system prompt size, (c) reduce max_tokens on driver.

### Data model (committed)

Minimal V1 schema. Everything else is computed.

```sql
-- The curriculum (seeded, CEFR-agnostic)
CREATE TABLE curriculum_items (
  id TEXT PRIMARY KEY,                    -- "sep_verbs", "perfectum", "topic_ordering_coffee"
  cefr_level TEXT NOT NULL,               -- "A1", "A2", "B1", etc.
  kind TEXT NOT NULL,                     -- "grammar_concept" | "topic" | "vocab_cluster"
  label TEXT NOT NULL,                    -- "separable verbs"
  description TEXT,                       -- teaching notes for the LLM
  prereq_ids TEXT                         -- JSON array of other item IDs
);

-- What the learner has done
CREATE TABLE learner_items (
  item_id TEXT PRIMARY KEY REFERENCES curriculum_items(id),
  attempts INTEGER DEFAULT 0,
  successes INTEGER DEFAULT 0,
  last_seen_at TIMESTAMP,
  confidence REAL DEFAULT 0,              -- 0..1, updated by post-session analyzer
  error_notes TEXT                        -- free-text observations, rolling window
);

-- Session transcripts (for post-session analysis + future review)
CREATE TABLE sessions (
  id TEXT PRIMARY KEY,
  started_at TIMESTAMP,
  ended_at TIMESTAMP,
  target_item_id TEXT,                    -- what the app picked to focus on today
  transcript_json TEXT,                   -- full back-and-forth
  analysis_json TEXT,                     -- post-session analyzer's output
  prompt_log_json TEXT                    -- [{turn_index, system_prompt, response}] for replay/debug
);

-- Anti-repetition: one row per buddy phrase uttered (Issue 1D)
CREATE TABLE buddy_phrases (
  id INTEGER PRIMARY KEY AUTOINCREMENT,
  session_id TEXT NOT NULL REFERENCES sessions(id),
  phrase_normalized TEXT NOT NULL,        -- lowercased, whitespace-normalized buddy line
  ts TIMESTAMP NOT NULL
);
CREATE INDEX idx_buddy_phrases_ts ON buddy_phrases(ts DESC);

-- Curriculum ordering (Issue 1F — precomputed topological order)
-- ALTER TABLE curriculum_items ADD COLUMN seed_order INTEGER NOT NULL;
-- (seeded at content-load time; used for cold-start target selection)
```

That's it. No SRS schedule table (stubbed as "resurface if last_seen > 1 session ago and confidence < 0.7"). No user table (single user). No topic table beyond curriculum_items.

**JSON shapes:**

- `transcript_json`: `[{role: "buddy" | "user", text: string, ts: ISO8601}]`
- `analysis_json`: `{item_id: string, turn_count: number, successes: number, attempts: number, errors: [{turn_index: number, error_type: string, note: string}], vocab_used_correctly: string[], vocab_struggled: string[]}`
- `learner_items.error_notes`: JSON array, cap 10 entries FIFO (newest first): `[{session_id, note, ts}]`

**Migrations:** keep a `migrations/` folder with numbered SQL files (`001_init.sql`, `002_add_b2_content.sql`). Apply in order on boot. Simple and obvious — no ORM needed for this scale.

### Session loop (committed)

A "turn" = one user utterance + one buddy reply.

1. **Open app.** Middleware validates session cookie (else redirect to `/login`). Server picks today's target:
   - **Cold start** (first session, `learner_items` empty): `SELECT * FROM curriculum_items ORDER BY seed_order ASC LIMIT 1`. For V1 seed, `topic_small_talk_greetings` has `seed_order = 1`.
   - **Warm path:** `SELECT * FROM learner_items WHERE last_seen_at < datetime('now', '-12 hours') AND confidence < 0.7 ORDER BY last_seen_at ASC NULLS LAST, confidence ASC LIMIT 1`. If nothing qualifies, pick next-available unseen item from `curriculum_items` respecting prereqs AND CEFR gating (A2 items blocked until confidence ≥ 0.6 on ≥4 A1 items).
2. **Show memory greeting.** Plain English. String is **composed in code from a template**, not via LLM call (for latency and determinism). Template: `"Last time you {last_outcome_phrase}. Today let's {today_plan}."`
3. **Tap-to-talk.** First user gesture calls `new AudioContext().resume()` (required — Safari blocks TTS playback without a user gesture) AND requests mic permission. App plays opening Dutch line via streaming TTS. For STT: browser requests a short-lived JWT from `/api/deepgram/token`, opens WSS directly to Deepgram, streams Opus-in-WebM chunks from MediaRecorder. Final transcript arrives via Deepgram's SpeechFinal event.
4. **Turn loop.** App builds system prompt per turn:
   - `persona` (fixed string: patient Dutch buddy, A1/A2 pacing — short sentences, high-frequency vocab, mostly present and perfectum, stretches to A2 structures when user handles them, corrects gently in reply, never mid-sentence)
   - `today's target` (item ID + label + teaching notes from `curriculum_items`)
   - `learner error notes` (last 3 `error_notes` entries for this item)
   - `last 6 turns` (3 exchanges, sliding window)
   - `anti-repetition window` (last 20 rows from `buddy_phrases`, ordered by ts DESC — O(1) indexed read, per Issue 1D)

   Calls Claude Sonnet 4.6 with streaming. Plays reply via streaming TTS. On completion, appends one row to `buddy_phrases` AND one entry to `prompt_log_json` array in `sessions.prompt_log_json` (Issue 1E — enables replay/debug).
5. **End after 7 turns OR 5 minutes, whichever comes first.** Hard cap. App shows "what Buddy noticed" panel in English with an **approximate rule-based summary** immediately (vocab seen count, turn count, target item label). Panel has a subtle "analyzing..." indicator while the async analyzer runs.
6. **Post-session analyzer (ASYNC via Vercel `waitUntil`).** After step 5 has already returned to the client, a background task runs a Claude call on the full transcript: "Did the user attempt {target item}? How many times? How many successes? List specific errors." Returns structured JSON matching `analysis_json` schema. Updates `learner_items.confidence` ONLY if `attempts > 0`. **Update rule (Issue 2A — Beta-prior smoothing):** `smoothed_rate = (successes + 2) / (attempts + 4)`, then weighted moving average `new_conf = 0.6 * old_conf + 0.4 * smoothed_rate`. Appends to `error_notes` (FIFO cap 10). When analyzer finishes, client polls or subscribes (Server-Sent Events) to update the panel with richer notes.
7. **Done.** Session row saved with transcript + analysis + prompt log. Next open, loop repeats with updated state.

### Error handling

- **STT connection fails mid-turn:** show "Having trouble hearing you — tap to try again." Retry at most twice, then end session gracefully (save partial transcript).
- **TTS fails:** fall back to showing Dutch text on screen with a "Tap to read aloud" retry. Log for investigation.
- **Claude API fails:** show "Buddy is thinking too hard — give it a moment" and retry once with exponential backoff. If it fails twice, end session and save what we have.
- **Mic permission denied:** onboarding screen explains the app needs mic access and links to Safari/Chrome permission settings. App cannot function without mic.
- **iOS AudioContext locked:** first tap-to-talk also calls `new AudioContext().resume()` to unlock audio playback for the session.
- **iOS AudioContext re-locked mid-session (V1 scope per eng review):** register a `document.addEventListener('visibilitychange', ...)` that calls `audioContext.resume()` whenever the tab becomes visible again. Handles the common case of user switching apps mid-conversation and returning — prevents silent playback failure after the return.

### The seeded curriculum for V1

20 items is enough to get a week of variety without committing to the full A1→A2 coverage up front. The set is weighted toward A1 consolidation (things he probably half-knows from Duolingo but hasn't *spoken*) with clear A2 stretch items. B1 concepts are deferred — they come in when confidence on the A2 core is earned.

**Grammar concepts (8):**
- Present tense — regular + high-frequency irregulars (zijn, hebben, gaan, komen) — **A1**
- Possessive pronouns (mijn/jouw/zijn/haar/ons/jullie/hun) — **A1**
- Word order in main clauses (V2 rule, time-manner-place) — **A1→A2**
- Modal verbs (kunnen/willen/moeten/mogen) — **A1→A2**
- Future with "gaan" — **A1→A2**
- Perfectum (het heb/is + voltooid deelwoord) — **A2**
- Separable verbs (opstaan, meegaan, uitgaan) — **A2**
- "Er" as expletive ("er is", "er zijn") — **A2** *(reserve bijzin word order and die/dat for a later content drop)*

**Topic vocab clusters (12):**
- Greetings & introductions (hoi/hallo, naam, waar kom je vandaan) — **A1**
- Numbers, dates, time (uren, dagen, afspraken) — **A1**
- Ordering at a café (koffie, thee, rekening) — **A1**
- Grocery shopping (Albert Heijn basics, prices, quantities) — **A1**
- Weather chat (het regent, koud, zonnig) — **A1**
- Describing your day (morgens, middag, avond + present tense) — **A1→A2**
- Describing your home (kamers, meubels, waar woon je) — **A1→A2**
- Small talk with neighbors (hoe gaat het, lekker weekend) — **A1→A2**
- Public transport (trein, tram, OV-chipkaart, stations) — **A1→A2**
- Giving and asking for directions (links, rechts, rechtdoor) — **A2**
- Making an appointment (afspraak maken, kapper, dokter) — **A2**
- Narrating your weekend using perfectum (wat heb je gedaan) — **A2**

Each item has an LLM teaching note + 3-5 opening-line templates. Items tagged with a CEFR level in `curriculum_items.cefr_level` so the app can gate access (only surface `A2` items once user has confidence ≥ 0.6 on at least 4 `A1` items).

## Open Questions

- **Exact pacing of a turn.** Does the buddy speak at native A1/A2 pace or slightly slowed? Native feels more authentic; slowed is easier to parse. Tunable via prompt ("speak at a pace suitable for an A1 learner pushing toward A2") — iterate once the loop is running.
- **Is this deployed publicly (even to a handful of friends learning Dutch) at V2, or strictly personal forever?** Affects auth decision down the road but not V1.

## Decided (was open)

- **Voice choice:** OpenAI TTS `nova` for V1. ElevenLabs A/B deferred — only reconsider if nova proves annoying after a week.
- **Correction style:** Errors surface in the post-session "What Buddy noticed" panel, mid-conversation stays flowing. The UI sketch shows this already.

## Success Criteria

V1 is successful if, after 7 consecutive days of real morning use, Adir can honestly say yes to all of:

1. "The app remembered something specific about yesterday and it felt good."
2. "I spoke Dutch out loud at least once per session and was understood at least 70% of the time."
3. "I would be sad if this app disappeared tomorrow."
4. "I'm using this INSTEAD of some of my Duolingo time, not on top of it."

Anti-success: if after 7 days Adir is skipping sessions or falling back to Duolingo out of frustration, the design is wrong. Iterate before adding features.

## Distribution Plan

V1: deploy to Vercel on a personal subdomain (e.g., `dutch.ronron.nl` or similar). Bookmark to home screen on phone. No accounts, no marketing. PWA manifest deferred to V1.1 — bookmark is enough to prove the loop.

Future (V2+): if it's working for Adir and friends ask to try it, add lightweight multi-user (magic link email auth, per-user learner model). Only if demand appears organically — do not build this speculatively.

CI/CD: Vercel auto-deploy on push to main. No staging env needed at this scale.

## Failure Modes (per new codepath)

| Codepath | Failure mode | Test? | Error handling? | User visibility |

| Post-session analyzer | Claude returns malformed JSON | Yes | Retry x1 with stricter prompt; on 2nd fail skip this session | Silent (panel stays approximate) |

## The Assignment (pre-build, revised after eng review)

Three things to do BEFORE any code, in order:

1. **STT reality check (30 min, Deepgram playground).** Pick 10 Dutch sentences spanning A1/A2 vocab and grammar (examples: "Ik heb gisteren naar de winkel gegaan," "Mijn buurman is heel aardig," "Kun je me helpen met mijn Nederlands"). Record yourself saying each one in Deepgram's playground with Nova-2 nl. Compute WER vs ground truth (counted errors / total words). If WER ≤ 25%, architecture is viable. If WER > 25%, pause before building and let's redesign the voice stack.

2. **Competitor field trip (30 min, Talkpal or similar).** Spend 30 full minutes speaking Dutch to Talkpal on your actual phone. Write down the three most concrete things you wish it did that it didn't. Bring that list to the weekend build — it will sharpen V1 scope in ways this review can't.

3. **LLM register spike (15 min, Claude console).** Paste 3 sample system prompts (cold-start + mid-session + post-session analyzer) into Claude Sonnet 4.6 and see if output lands at A1/A2 register. Calibrate the persona string before writing code — it's cheaper to iterate the prompt in the console than in the app.

Total pre-build cost: ~75 min. Saves potentially 16-24h of building on a voice stack that doesn't understand your Dutch.
